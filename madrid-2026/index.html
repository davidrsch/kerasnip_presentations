The Landscape

🧱 Keras (via keras3)

  • Deep Learning
  • Tensor-centric & Imperative
  • Backend agnostic (TF, JAX, Torch)
  • Arbitrary graphs of layers

🧹 Tidymodels

  • Machine Learning
  • Dataframe-centric & Declarative
  • Unified interface (fit, predict)
  • Standardized tuning

The Gap: Using tune_grid() to optimize Neural Network topology (depth/width) is difficult because parsnip expects fixed arguments, while Keras models are arbitrary graphs.

Goal: contrast the two mental models (code vs workflow).

Say: - Keras is great for building; tidymodels is great for workflow. - The gap is in architectural tuning.

Plain-language framing: - In tidymodels, a “model spec” is basically a function call with named arguments. - In Keras, the architecture is a graph you build with code. - So you can tune things that already exist as arguments (learning rate), but changing topology usually means rewriting the model code.

Bridge we want: - Make architecture decisions look like spec arguments so tune_grid() can control them.

The Pain Point (in one sentence)

Keras models are arbitrary graphs, but parsnip specs are fixed argument sets.

So you can tune learning rate easily, but tuning model structure (depth / width / branches) is awkward.

Goal: make the problem feel concrete and familiar.

Say: - This is not “deep learning 101”. It’s: how to keep Keras inside the tidymodels workflow. - The friction is that architecture changes live in Keras code, but tidymodels tuning expects named arguments.

The Solution: Dynamic Specs

kerasnip is a Model Builder, not a Model Zoo.

  1. Define Blocks: Functions returning Keras layers.
  2. Meta-program Spec: create_keras_*_spec() generates a new parsnip model.
  3. Tune Structure: Block arguments (and repetition count) can be tuned.

Goal: explain the “dynamic spec” idea in one breath.

Say: - This is a code-generation approach: you define blocks, then kerasnip generates a parsnip spec function. - Once it’s a parsnip spec, the rest is standard: recipes → workflows → tune_grid().

Naming Conventions (auto-generated args)

  • Arguments get prefixed by block name (e.g. body_units)
  • Repeatable blocks expose num_{block} (e.g. num_body)
  • Repetitions share the same hyperparameters (by design)

Example mapping:

dense_block(model, units = 128, dropout = 0.1) → body_units, body_dropout

Goal: remove naming confusion before we touch code.

Say: - Block args become spec args with prefixes (e.g. body_units). - Repetition is controlled by num_{block}. - Repeated blocks share hyperparameters by design.

Sequential vs Functional (rule of thumb)

  • Sequential: one stream of features, simple stacks
  • Functional: multi-input, branch/merge, skip connections, explicit graphs

Goal: give a decision rule people can remember.

Say: - Use sequential for “a chain of layers”. - Use functional when you need an explicit graph: branches/merges, skips, multi-input.

Common Gotchas

  • Ensure the recipe outputs numeric matrices (dummy + normalize)
  • Start with a small grid, confirm structure via compile_keras_grid()
  • Keep tuning spaces reasonable (depth × width can explode)

Extra gotchas to mention out loud: - Classification: make sure the outcome is a factor (tidymodels convention) and the final layer/loss matches binary vs multiclass. - Every grid point is a fresh model build + fresh training; set seeds if you want stable comparisons. - Training controls (epochs/callbacks) should be held constant during architecture search unless you intentionally tune them.

Sequential Blocks (minimal)

input_block <- function(model, input_shape) {
  keras_model_sequential(input_shape = input_shape)
}

dense_block <- function(model, units = 128, dropout = 0.0) {
  model |>
    layer_dense(units = units, activation = "relu") |>
    layer_dropout(rate = dropout)
}

output_block <- function(model) {
  model |> layer_dense(units = 1)
}

Goal: show that blocks are simple and reusable.

Say: - Blocks are just functions. - The signature matters: args become spec args (prefixed by block name).

Good practice: - Keep blocks deterministic. - Use sensible defaults so the base model runs without tuning.

If asked “can I make each repeated layer different?” - Repetitions share hyperparameters by design. - If you want different parameters per layer, define multiple blocks (e.g. body1, body2) instead of repeating one.

Transition: Next: generate the spec from these blocks.

Generate the parsnip spec

create_keras_sequential_spec(
  model_name = "diamond_mlp",
  layer_blocks = list(
    input = input_block,
    body = dense_block,  # <--- This block will be repeatable
    output = output_block
  ),
  mode = "regression"
)

Goal: make the “spec generation” moment land.

Say: - This creates a new parsnip spec function: diamond_mlp(). - From this point on, it behaves like any other tidymodels model: you can set engine, add to workflows, and tune args.

Transition: Next we’ll tune topology via num_body.

Structural Tuning

We treat topology as a hyperparameter.

spec <- diamond_mlp(
  num_body = tune(),       # <--- Depth (e.g., 1, 2, or 3 layers)
  body_units = tune(),     # <--- Width (e.g., 64 vs 128 units)
  body_dropout = tune(),
  fit_epochs = 20
) |>
  set_engine("keras")

The arguments num_{block} are automatically generated by kerasnip for any repeatable block.

Goal: make “depth is tunable” feel obvious.

Say: - num_body allows searching topology inside a standard tune_grid().

What to emphasize: - Depth becomes a first-class parameter. - Keep training controls mostly fixed during architecture search.

Practical tip: - Tune structure first (depth/width), then optionally tune optimizer/training hyperparameters.

Diamonds + Tidymodels Workflow

library(tidymodels)
library(kerasnip)

data(diamonds, package = "ggplot2")

diamonds_split <- initial_split(diamonds, strata = price)
diamonds_train <- training(diamonds_split)

diamonds_rec <- recipe(price ~ ., data = diamonds_train) |>
  step_dummy(all_nominal_predictors()) |>
  step_zv(all_predictors()) |>
  step_normalize(all_numeric_predictors())

wf <- workflow() |>
  add_recipe(diamonds_rec) |>
  add_model(spec)

Tuning Grid (including depth)

params <- extract_parameter_set_dials(wf) |>
  update(
    num_body     = dials::num_terms(c(1, 4)),
    body_units   = dials::hidden_units(c(32, 256)),

res   <- tune_grid(wf, resamples = folds, grid = grid)

Functional Wiring with inp_spec()

inp_spec(func, inputs) explicitly maps upstream block outputs to function arguments.

That’s the key to: - multi-input models - skip connections - branch/merge architectures

Goal: explain inp_spec() as readable wiring.

Say: - Most people get functional Keras, but the missing piece is: expressing the graph cleanly in R.

Rule of thumb: - If you can describe your model as a simple chain, use sequential. - Use functional when you need named tensors: branches/merges, shared towers, skip connections, or multiple inputs.

Also worth saying: - inp_spec() is the wiring layer: it makes the graph explicit and readable.

Transition: Next: minimal functional blocks.

Functional Blocks (minimal)

# In functional Keras, tensors flow through your blocks.
input_block <- function(input_shape) {
  layer_input(shape = input_shape, name = "features")
}

dense_block <- function(tensor, units = 64, dropout = 0.0) {
  tensor |>
    layer_dense(units = units, activation = "relu") |>
    layer_dropout(rate = dropout)
}

concat_block <- function(input_a, input_b) {
  layer_concatenate(list(input_a, input_b))
}

output_block <- function(tensor, num_classes) {
  tensor |> layer_dense(units = num_classes, activation = "softmax")
}

Goal: show block conventions and avoid wiring mistakes.

Say: - Two key correctness points (from the kerasnip docs):

  • The input block is function(input_shape) (no model argument).
  • Downstream blocks take the upstream tensor as the first argument.

inp_spec() is how we wire outputs into named arguments.

Common pitfall: - The names you reference in inp_spec(..., "main_input") must match block names exactly.

Transition: Next: generate the functional spec.

Build the graph (spec)

create_keras_functional_spec(
  model_name = "attrition_towers",
  layer_blocks = list(
    main_input = input_block,
    tower_a = inp_spec(dense_block, "main_input"),
    tower_b = inp_spec(dense_block, "main_input"),
    joined = inp_spec(concat_block, c(input_a = "tower_a", input_b = "tower_b")),
    output = inp_spec(output_block, "joined")
  ),
  mode = "classification"
)

Goal: show functional graphs still fit a standard workflow.

Say: - This gives us a non-linear graph while still using a standard tidymodels recipe.

If your architecture is a straight line, sequential is simpler. Functional is for “graphs”: branches, merges, skips.

If asked about multi-input: - Define multiple layer_input() blocks and wire them into downstream blocks by name using inp_spec().

Practical Tips (so you don’t suffer)

  • Start with compile_keras_grid() before tuning
  • Keep blocks small and testable
  • Prefer a few meaningful tunables over 30 tiny ones

Goal: give survival advice people can apply tomorrow.

Say: - If something breaks in the functional graph, the error messages can be deep in Keras. - Reducing the graph to 2–3 blocks is the fastest debugging strategy.

More “power user” tips to mention verbally: - If you need a different predictor format, consider step_collapse() to bundle predictors into a single list-column (useful for custom model interfaces). - For custom optimizers/losses/metrics, register them: register_keras_optimizer(), register_keras_loss(), register_keras_metric(). - Use the discovered Keras object lists (keras_optimizers, keras_losses, keras_metrics) to keep names valid across Keras versions.

Advantages & Limitations

Advantages

  1. Workflow Parity: Use recipes, workflows, and tune exactly like glmnet.
  2. Reproducibility: Topology is code. Version control your architecture.
  3. Extensibility: Wrap any Keras layer (Transformers, RNNs) into a block.

Limitations

  • Manual Blocking: No pre-shipped “ResNet50” blocks yet (you must define them).
  • Custom Losses: Integration requires some boilerplate for custom objects.

Goal: be honest about trade-offs while keeping the pitch credible.

Say: - The learning curve is “how to write good blocks”. - Once blocks exist, the workflow becomes familiar again.

If someone asks “how do I evaluate / inspect the Keras object?” - extract_keras_model() returns the underlying Keras model. - extract_keras_history() returns training history. - keras_evaluate() can run evaluation on provided x/y.

If asked about templates: the goal is to add helpers without turning the package into a model zoo.
