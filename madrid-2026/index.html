The Landscape

🧱 Keras (via keras3)

  • Deep Learning
  • Tensor-centric & Imperative
  • Backend agnostic (TF, JAX, Torch)
  • Arbitrary graphs of layers

🧹 Tidymodels

  • Machine Learning
  • Dataframe-centric & Declarative
  • Unified interface (fit, predict)
  • Standardized tuning

The Gap: Using tune_grid() to optimize Neural Network topology (depth/width) is difficult because parsnip expects fixed arguments, while Keras models are arbitrary graphs.

Keras is great for building; Tidymodels is great for workflow. The gap is in architectural tuning.

The Solution: Dynamic Specs

kerasnip is a Model Builder, not a Model Zoo.

  1. Define Blocks: Functions returning Keras layers.
  2. Meta-program Spec: create_keras_*_spec() generates a new parsnip model.
  3. Tune Structure: Block arguments (and repetition count) can be tuned.

Workflow 1: Sequential API (Regression)

Goal: Predict Diamond Prices. Tune architecture depth and width.

# 1. Define a "Body Block" that can be repeated
dense_block <- function(model, units = 128) {
  model |> layer_dense(units = units, activation = "relu")
}

# 2. Generate the parsnip model function 'diamond_mlp'
create_keras_sequential_spec(
  model_name = "diamond_mlp",
  layer_blocks = list(
    input = input_block,
    body = dense_block,  # <--- This block will be repeatable
    output = output_block
  ),
  mode = "regression"
)

Structural Tuning

We treat topology as a hyperparameter.

# 3. Define the tunable specification
spec <- diamond_mlp(
  num_body = tune(),       # <--- Tunes Depth (e.g., 1, 2, or 3 layers)
  body_units = tune(),     # <--- Tunes Width (e.g., 64 vs 128 units)
  fit_epochs = 20
) |>
  set_engine("keras")

The arguments num_{block} are automatically generated by kerasnip for any block in the sequential spec.

Key value prop: num_body allows searching topology inside a standard grid_search.

Workflow 2: Functional API (Multi-Input)

Goal: Employee Attrition (Classification). - Branch A: Personal (Age, Distance, Education). - Branch B: Work (JobRole, OverTime, Dept).

create_keras_functional_spec(
  model_name = "attrition_net",
  layer_blocks = list(
    # Inputs & Branches
    personal_input = input_pers_block,
    work_input = input_work_block,
    
    proc_personal = inp_spec(dense_block, "personal_input"),
    proc_work     = inp_spec(dense_block, "work_input"),
    
    # Join & Output
    joined = inp_spec(omit_concat, c(a="proc_personal", b="proc_work")),
    output = inp_spec(out_block, "joined")
  ),
  mode = "classification"
)

inp_spec(func, inputs) explicitly maps outputs of one block to arguments of another.

We use Attrition dataset here (different from paper’s Ames). Processing personal vs work info separately lets the network learn distinct representations before merging.

Advantages & Limitations

Advantages

  1. Workflow Parity: Use recipes, workflows, and tune exactly like glmnet.
  2. Reproducibility: Topology is code. Version control your architecture.
  3. Extensibility: Wrap any Keras layer (Transformers, RNNs) into a block.

Limitations

  • Manual Blocking: No pre-shipped “ResNet50” blocks yet (you must define them).
  • Custom Losses: Integration requires some boilerplate for custom objects.

Roadmap

  • 📦 Templates: High-level helpers for common architectures (CNN/Transformers).
  • 💾 tfdatasets: Native integration for out-of-memory data loading.
  • 🔧 Fine-tuning: Workflows for adapting pre-trained models.
