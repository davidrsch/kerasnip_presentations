El Panorama

🧱 Keras (vía keras3)

  • Aprendizaje profundo
  • Pensado para tensores
  • Agnóstico al backend (TF, JAX, Torch)
  • Grafos arbitrarios de capas

🧹 tidymodels

  • Aprendizaje automático
  • Pensado para data frames
  • Interfaz unificada (fit, predict)
  • Ajuste de hiperparámetros, de forma estandarizada

La brecha: optimizar la topología (profundidad/ancho) con tune_grid() es difícil: parsnip espera un conjunto fijo de argumentos, mientras que Keras permite grafos arbitrarios.

Objetivo: contrastar “código” vs “workflow”.

Qué decir: - Keras es excelente para construir; tidymodels, para el flujo. - La brecha está en el ajuste arquitectónico.

En sencillo: - En tidymodels, un “spec” es básicamente una llamada a función con argumentos. - En Keras, la arquitectura es un grafo que se construye con código. - Por eso se ajusta bien lo que ya es argumento (learning rate), pero cambiar topología suele implicar reescribir el modelo.

Puente que buscamos: - Convertir decisiones de arquitectura en argumentos ajustables para que tune_grid() los controle.

Transición: Siguiente: el enfoque de specs dinámicos.

El Problema (en una frase)

Los modelos Keras son grafos arbitrarios, pero los specs de parsnip son conjuntos fijos de argumentos.

El learning rate se ajusta fácilmente, pero ajustar la estructura (profundidad/ancho/ramas) suele ser incómodo.

Objetivo: que el problema se entienda en 10 segundos.

Qué decir: - Esta charla no es “deep learning 101”. Es: cómo mantener Keras dentro del flujo de tidymodels. - La fricción está en que la arquitectura vive en código Keras, pero el ajuste en tidymodels espera argumentos con nombre.

Transición: Entonces: ¿cómo convertimos decisiones de topología en argumentos ajustables?

La Solución: Specs Dinámicos

kerasnip es un constructor de modelos, no un “model zoo”.

  1. Definir bloques: funciones que devuelven capas Keras.
  2. Generar el spec: create_keras_*_spec() crea un modelo de parsnip.
  3. Ajustar la estructura: los argumentos (y el número de repeticiones) pasan a ser hiperparámetros.

Objetivo: explicar “specs dinámicos” sin jerga.

Qué decir: - Es un enfoque de generación de código: defines bloques y kerasnip genera un spec de parsnip. - A partir de ahí, todo vuelve a ser estándar: recipes → workflows → tune_grid().

Convenciones de nombres (args autogenerados)

  • Los argumentos se prefijan con el nombre del bloque (p. ej., body_units)
  • Los bloques repetibles exponen num_{block} (p. ej., num_body)
  • Las repeticiones comparten hiperparámetros (a propósito)

Ejemplo de mapeo:

dense_block(model, units = 128, dropout = 0.1) → body_units, body_dropout

Objetivo: eliminar confusión de nombres antes del código.

Qué decir: - Los args del bloque pasan a ser args del spec con prefijo (p. ej. body_units). - La repetición se controla con num_{block}. - Las repeticiones comparten hiperparámetros a propósito.

Transición: Con ese mapeo, el ejemplo secuencial se ve “normal”.

Secuencial vs Funcional (regla práctica)

  • Secuencial: un flujo de variables, apilados sencillos
  • Funcional: multi-entrada, ramas/uniones, conexiones de salto (skip connections), grafos explícitos

Objetivo: dar una regla práctica fácil de recordar.

Qué decir: - Secuencial para “una cadena de capas”. - Funcional cuando necesitas un grafo explícito: ramas/uniones, skip connections, multi-entrada.

Transición: Empezamos con secuencial (más simple) y luego volvemos a funcional.

Errores Comunes

  • Asegúrate de que la receta produzca matrices numéricas (dummy + normalize)
  • Empieza con un grid pequeño y confirma la estructura con compile_keras_grid()
  • Mantén el espacio de ajuste razonable (profundidad × ancho explota)

Objetivo: prevenir los fallos más comunes.

Qué decir: - Esta diapositiva está aquí para ahorrarte dolores en el turno de preguntas. - Clasificación: asegúrate de que el outcome sea factor (convención tidymodels) y que la última capa/loss encaje en binaria vs multiclase. - Cada punto del grid es construir + entrenar un modelo nuevo; fija seeds si quieres comparaciones estables.

Flujo 1: API Secuencial (Regresión)

Objetivo: Predecir precios de diamantes y ajustar profundidad + ancho.

Dataset: ggplot2::diamonds.

Objetivo: un demo simple pero creíble.

Qué decir: - MLP tabular con diamonds. Simple a propósito: el foco es ajustar profundidad/ancho, no exprimir el feature engineering.

Transición: Primero definimos bloques (funciones), luego generamos el spec.

Bloques Secuenciales (mínimos)

input_block <- function(model, input_shape) {
  keras_model_sequential(input_shape = input_shape)
}

dense_block <- function(model, units = 128, dropout = 0.0) {
  model |>
    layer_dense(units = units, activation = "relu") |>
    layer_dropout(rate = dropout)
}

output_block <- function(model) {
  model |> layer_dense(units = 1)
}

Objetivo: que se vea que los bloques son simples y reutilizables.

Qué decir: - Los bloques son funciones normales; la firma importa porque se transforma en argumentos del spec.

Buenas prácticas: - Mantén los bloques deterministas. - Usa valores por defecto razonables para que el modelo base funcione sin ajuste.

Si preguntan “¿puedo tener parámetros distintos por capa?” - Las repeticiones comparten hiperparámetros a propósito. - Si quieres diferencias por capa, define varios bloques (body1, body2, …) en vez de repetir uno.

Transición: Siguiente: generar el spec a partir de estos bloques.

Generar el spec parsnip

create_keras_sequential_spec(
  model_name = "diamond_mlp",
  layer_blocks = list(
    input = input_block,
    body = dense_block,  # <--- Bloque repetible
    output = output_block
  ),
  mode = "regression"
)

Objetivo: que el “momento generación de spec” se entienda.

Qué decir: - Esto crea una nueva función spec de parsnip: diamond_mlp(). - A partir de aquí se comporta como cualquier modelo de tidymodels: engine, workflow y tuning.

Transición: Siguiente: ajustar topología con num_body.

Ajuste Estructural

Tratamos la topología como hiperparámetro.

spec <- diamond_mlp(
  num_body = tune(),       # <--- Profundidad (ej. 1, 2, o 3 capas)
  body_units = tune(),     # <--- Ancho (ej. 64 vs 128 unidades)
  body_dropout = tune(),
  fit_epochs = 20
) |>
  set_engine("keras")

Los argumentos num_{block} son generados automáticamente por kerasnip para cualquier bloque repetible.

Objetivo: que num_body se entienda como hiperparámetro de topología.

Qué decir: - Valor clave: num_body permite buscar topología dentro de un tune_grid() estándar. - La profundidad pasa a ser un hiperparámetro más. - Durante la búsqueda arquitectónica, conviene mantener el entrenamiento bastante fijo. - Primero estructura (profundidad/ancho), luego (si hace falta) hiperparámetros de entrenamiento.

Flujo 2: API Funcional (Ramificar/Unir)

Meta: Predecir rotación de personal (attrition) — clasificación.

Dataset: modeldata::attrition.

Construimos un grafo sencillo de ramas y unión: - una entrada - dos “torres” en paralelo - concat + salida

Objetivo: motivar por qué el API funcional importa.

Qué decir: - El valor del API funcional es que dejas de estar limitado a una cadena lineal. - Decisión práctica para que sea reproducible: usamos UNA entrada desde una receta estándar y creamos dos ramas desde el mismo tensor.

Transición: inp_spec() es el mecanismo de cableado.

Conexiones con inp_spec()

inp_spec(func, inputs) mapea explícitamente salidas de bloques a argumentos.

Útil para: - modelos multi-entrada - skip connections - arquitecturas con ramas y uniones

Objetivo: explicar inp_spec() como cableado legible.

Qué decir: - La mayoría entiende el API funcional, pero necesita una forma clara de expresarlo en R. - Regla práctica: si tu modelo es una cadena simple, secuencial; si necesitas tensores con nombre (ramas/uniones, torres compartidas, conexiones de salto, multi-entrada), funcional. - inp_spec() es la capa de cableado: hace el grafo explícito y legible.

Transición: Siguiente: bloques funcionales mínimos.

Bloques Funcionales (mínimos)

# En Keras funcional, los tensores van “pasando” por bloques.
input_block <- function(input_shape) {
  layer_input(shape = input_shape, name = "features")
}

dense_block <- function(tensor, units = 64, dropout = 0.0) {
  tensor |>
    layer_dense(units = units, activation = "relu") |>
    layer_dropout(rate = dropout)
}

concat_block <- function(input_a, input_b) {
  layer_concatenate(list(input_a, input_b))
}

output_block <- function(tensor, num_classes) {
  tensor |> layer_dense(units = num_classes, activation = "softmax")
}

Objetivo: evitar errores típicos de cableado.

Qué decir: - Dos puntos de corrección (según la documentación de kerasnip): el bloque de entrada es function(input_shape) (sin argumento model) y los bloques posteriores reciben el tensor upstream como primer argumento. - inp_spec() es lo que nos permite cablear salidas a argumentos con nombre. - Error típico: los nombres usados en inp_spec(..., "main_input") tienen que coincidir exactamente con los nombres de bloque.

Transición: Siguiente: generar el spec funcional.

Construir el grafo (spec)

create_keras_functional_spec(
  model_name = "attrition_towers",
  layer_blocks = list(
    main_input = input_block,
    tower_a = inp_spec(dense_block, "main_input"),
    tower_b = inp_spec(dense_block, "main_input"),
    joined = inp_spec(concat_block, c(input_a = "tower_a", input_b = "tower_b")),
    output = inp_spec(output_block, "joined")
  ),
  mode = "classification"
)

Objetivo: mostrar que un grafo sigue encajando en tidymodels.

Qué decir: - Con esto tenemos un grafo no lineal sin complicarnos con múltiples entradas desde la receta. - Si tu arquitectura es una cadena, el API secuencial suele ser más simple; funcional es para “grafos”: ramas, uniones, skip connections. - Si preguntan por multi-entrada: define varios bloques layer_input() y conéctalos por nombre con inp_spec().

Consejos Prácticos

  • Empieza con compile_keras_grid() antes de ajustar
  • Mantén bloques pequeños y testeables
  • Pocos hiperparámetros importantes > muchos microajustes

Objetivo: dar consejos que eviten sufrimiento mañana.

Qué decir: - Si algo falla en el grafo funcional, reduce a 2–3 bloques para aislar el error. - Si necesitas un formato distinto de predictores, considera step_collapse() para agrupar predictores en una única columna tipo lista. - Para optimizadores/losses/métricas custom: register_keras_optimizer(), register_keras_loss(), register_keras_metric(). - Para evitar nombres inválidos según tu versión de Keras: usa keras_optimizers, keras_losses, keras_metrics.

Transición: Cierro con ventajas/limitaciones y hoja de ruta.

Ventajas y Limitaciones

Ventajas

  1. Paridad: Usa recipes, workflows y tune igual que con glmnet.
  2. Reproducibilidad: La topología es código. Control de versiones total.
  3. Extensibilidad: Envuelve cualquier capa Keras (Transformers, RNNs) en un bloque.

Limitaciones

  • Bloques manuales: no hay bloques tipo “ResNet50” prehechos (hay que definirlos).
  • Funciones de pérdida: una integración profunda de pérdidas personalizadas requiere algo de código repetitivo.

Objetivo: ser honesto con los límites sin “pinchar” la propuesta.

Qué decir: - El trabajo está en definir buenos bloques. - Después, el flujo vuelve a ser familiar.

Si alguien pregunta por evaluación/inspección del modelo: - extract_keras_model() devuelve el modelo Keras subyacente. - extract_keras_history() devuelve el historial de entrenamiento. - keras_evaluate() permite evaluar con x/y.

Sobre plantillas: la idea es añadir helpers sin convertir el paquete en un “model zoo”.
